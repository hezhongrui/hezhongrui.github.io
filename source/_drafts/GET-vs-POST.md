    <article class="markdown-body entry-content" itemprop="text"><p>title: GET vs POST
date: 2016-04-11 13:54:12</p>

<h2><a id="user-content-desc-辨析-http-中-get-请求和-post-请求的差异" class="anchor" href="#desc-辨析-http-中-get-请求和-post-请求的差异" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>desc: 辨析 HTTP 中 get 请求和 post 请求的差异</h2>

<p>发起 HTTP 请求有四种基本方法：PUT、DELETE、POST 和 GET，通俗讲可以认为对应于增删改查四个方面。根据 HTTP 规范，GET 请求用于获取信息，是安全和幂等的，这里的安全并不代表不会泄露信息，而是说它用来获取信息，查询数据库，不会影响数据库的数据，也不会影响资源的状态；POST 请求则有可能会修改服务器上数据库的数据和资源的状态。</p>

<p>对于 <code>&lt;form&gt;&lt;/form&gt;</code> 标签，其 <code>action</code> 属性的默认值是 GET，可以修改为 POST。使用 GET 时，表单数据中的数据会被转义并以查询字符串的形式追加到 URL 的尾部；使用 POST 时，表单数据会依附请求体发送给服务器。</p>



<p>下面的表格从多方面比较了两者的差异：</p>

<table><thead>
<tr>
<th>维度</th>
<th>GET（HTTP）</th>
<th>POST（HTTP）</th>
</tr>
</thead><tbody>
<tr>
<td>历史记录</td>
<td>由于数据被添加到了 URL 中，所以会保留在浏览器的历史记录之中</td>
<td>数据不会保留在浏览器的历史记录之中</td>
</tr>
<tr>
<td>书签</td>
<td>可收藏为书签</td>
<td>不可收藏为书签</td>
</tr>
<tr>
<td>后退和重新提交</td>
<td>如果 HTML 被浏览器缓存了，则重新执行 GET 请求可能不会向服务器重新提交数据</td>
<td>浏览器应该提醒用户：数据需要重新提交</td>
</tr>
<tr>
<td>编码格式</td>
<td>"application/x-www-form-urlencoded"</td>
<td>"multipart/form-data"（主要用于二进制数据）或 "application/x-www-form-urlencoded"</td>
</tr>
<tr>
<td>参数</td>
<td>长度受限于 URL 的长度，通常在 2K 以内，具体长度取决于浏览器和服务器</td>
<td>可发送参数，包括上传的文件等</td>
</tr>
<tr>
<td>是否可以修改</td>
<td>可被脚本修改</td>
<td>比较难修改</td>
</tr>
<tr>
<td>数据类型是否有限制</td>
<td>只接受 ASCII 字符，其他字符会被转义</td>
<td>无限制，可接收二进制数据</td>
</tr>
<tr>
<td>安全性</td>
<td>GET 相比于 POST 的安全性低，因为发送的数据在 URL 中可见，且可以被浏览器历史记录、服务器日志记录</td>
<td>POST 的参数不会被浏览器历史记录和服务器日记记录</td>
</tr>
<tr>
<td>数据长度限制</td>
<td>受限于 URL 的长度，通常小于 2K，具体限制取决于浏览器和服务器</td>

</tr>
<tr>

<td>可被缓存</td>
<td>不可被缓存</td>

<h2><a id="user-content-优点和缺点" class="anchor" href="#优点和缺点" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>优点和缺点</h2>

<p>当使用 GET 请求发送表单数据时：</p>

<ul>
<li>表单数据只能是 ASCII 字符，其他字符需要被转义为 ASCII 格式，但 POST 请求可以处理二进制数据、图片和其他文件</li>
<li>所有的表单数据对于 URL 都是可见的，而且会被缓存在浏览器的历史记录中，降低了安全性</li>
<li>GET 请求的一个优势是表单数据可以被收藏为书签，也可以直接用于填充表单数据</li>
<li>表单数据的长度会受到 URL 长度的限制</li>
<li>易于遭受脚本攻击</li>
</ul>

<h2><a id="user-content-服务器端" class="anchor" href="#服务器端" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>服务器端</h2>

<p>由于 GET 和 POST 对数据的编码方式不同，所以在服务端解码数据的方式也不同。在 ASP 中，服务端使用 <code>Request.QueryString</code> 获取 GET 请求参数，使用 <code>Request.Form</code> 获取 POST 请求参数。</p>

<h2><a id="user-content-https" class="anchor" href="#https" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>HTTPS</h2>

<p>使用 HTTPS 发起 GET 请求是否会更安全些呢？我们就做如下的请求来回答这一问题：</p>

<div class="highlight highlight-source-shell"><pre>GET https://www.example.com/login.php<span class="pl-k">?</span>user=mickey<span class="pl-k">&amp;</span>passwd=mini</pre></div>

<p>假设有人监视了这一 GET 请求，那么他只会看到如下信息：</p>

<ul>
<li>这是一个 HTTPS 链接</li>
<li>域名 <code>www.example.com</code></li>
<li>请求的长度</li>
<li>响应的长度</li>
</ul>

<p>URL 路径以及查询字符串都是不可见的。由此可见，在这些方面，GET 请求和 POST 请求的安全性相同。但是，浏览器仍然会记录 URL，所以无论是 HTTP 还是 HTTPS，使用 GET 发送敏感信息仍是不安全的。</p>
